## Architektur

```

                              вҶ’ /auth/login     вҶ’ redirect zu Google (OIDC)
                              вҶ’ /auth/callback  вҶ’ Token validieren, Session anlegen

                              вҶ’ /api/*          вҶ’ BookStack (Token aus req.session.bookstackToken, serverseitig)

Cron (tГӨglich 02:00) вҶ’ syncAllBooks() вҶ’ getAllUserTokens() вҶ’ page_stats + book_stats_history
```

**Auth-Flow:** Alle Routen ausser `/auth/*` sind durch einen Session-Guard geschГјtzt. Nicht authentifizierte Requests auf HTML-Pfade вҶ’ Redirect `/auth/login` вҶ’ Google вҶ’ `/auth/callback` вҶ’ Session вҶ’ `/`. API-Pfade (JSON-Requests) вҶ’ `401 JSON`.

KI-Aufrufe laufen **nicht direkt aus dem Browser**, sondern Гјber den jeweiligen Proxy-Endpunkt вҖ“ der Server hГӨlt alle Credentials. Der BookStack-Proxy injiziert den per-User-Token aus `req.session.bookstackToken` serverseitig; das Frontend sendet keinen Authorization-Header.

## Prompt-Konfiguration

**Pflichtdatei:** `prompt-config.json` (Projektroot) muss vorhanden sein. Fehlt sie, wirft der Server beim Start einen Fehler вҖ“ es gibt keine eingebauten Defaults. Struktur und Inhalt: siehe `prompt-config.json`.

**Hartkodiert in `prompts.js`:** Die JSON-Only-Anweisung, alle JSON-Schema-Definitionen und Feldnamen in den Build-Funktionen.

**Ladereihenfolge:**
- `routes/proxies.js` liest die Datei beim ersten `/config`-Aufruf (lazy singleton) und liefert sie als `promptConfig` ans Frontend.
- `routes/jobs.js` und `routes/chat.js` lesen die Datei beim Modulstart (synchron) und rufen `configurePrompts()` einmalig nach dem dynamischen `import()` von `prompts.js` auf.
- `public/js/app.js` ruft `configurePrompts(cfg.promptConfig)` in `init()` auf вҶ’ setzt die `export let SYSTEM_*`-Variablen in `prompts.js` via ESM-Live-Binding.

**Single Source of Truth:** Rollenformulierungen und Basisregeln in `prompt-config.json`; JSON-Schemas und Build-Logik aller Prompts ausschliesslich in `public/js/prompts.js`.

## KI-Integration

**Regel: Alle serverseitigen KI-Calls laufen Гјber die Job-Queue (`/jobs/*`).** Neue Features die einen KI-Call benГ¶tigen, implementieren einen Job-Typ in `routes/jobs.js` (Funktion `runXxxJob` + `router.post('/xxx', ...)`). Direkte synchrone KI-Calls aus Route-Handlern (ohne Job-Queue) sind nicht erlaubt вҖ“ auch nicht fГјr kurze/einfache Operationen. Das Frontend pollt den Job-Status via `_startPoll()`. Ausnahme: Seiten-Chat (`/chat/send`) nutzt bewusst SSE-Streaming statt Jobs.

**`callAI`** (in `lib/ai.js`, importiert von `routes/jobs.js`) gibt immer ein **geparstes JSON-Objekt** zurГјck вҖ“ kann nicht fГјr Freitext-Antworten verwendet werden. Jeder Systemprompt muss JSON-Only erzwingen (die `JSON_ONLY`-Konstante in `prompts.js`).

**Schema-Validierung:** Nach jedem `callAI`-Aufruf wird das Pflichtfeld geprГјft (`fehler` in `lektorat.js`, `gesamtnote` in `review.js`, `figuren` in `figuren.js`). Fehlt es, Fehler werfen statt falsche Daten rendern.

**Provider-Proxy:**
- **`/claude`** вҖ“ injiziert `ANTHROPIC_API_KEY`, leitet an `api.anthropic.com/v1/messages` weiter, streamt als SSE. `model` und `max_tokens` werden aus `.env` (MODEL_NAME / MODEL_TOKEN) erzwungen.
- **`/ollama`** вҖ“ konvertiert Anthropic-Request-Format вҶ’ Ollama `/api/chat`-Format, normalisiert NDJSON вҶ’ SSE. Frontend-Parser bleibt fГјr beide Provider identisch.

## Chat

**Seiten-Chat** (`/chat/send`): Direktes SSE-Streaming, kein Job-Queue. KI-Antwortformat: `{ "antwort": "Markdown", "vorschlaege": [{ "original", "ersatz", "begruendung" }] }`. `original` muss zeichengenau im Seitentext vorkommen. `applyChatVorschlag()` ersetzt es identisch zum Lektorat-Flow via `_applyCorrections`.

**Buch-Chat** (`/jobs/book-chat`): Job-Queue statt SSE, kein VorschlГӨge-System. Buch-Chat-Sessions nutzen `page_id=0` und `page_name='__book__'` in der `chat_sessions`-Tabelle. KI-Antwortformat: `{ "antwort": "Markdown" }`.

**SSE-Fehlerbehandlung:** Das Flag `sseStarted` trennt Fehler vor dem Stream-Start (вҶ’ JSON 502) von Fehlern wГӨhrend des Streams (вҶ’ SSE `{ type: 'error' }` + `[DONE]`). Express 4 fГӨngt Fehler aus `async`-Handlern nicht automatisch вҖ“ ohne den try/catch-Wrapper schliesst Node die Verbindung ohne HTTP-Antwort.

**`bsGetAll(path)`:** BookStack paginiert Listenendpunkte (Standard: 20 EintrГӨge). `bsGetAll` iteriert automatisch alle Seiten. Immer `bsGetAll` statt `bsGet` verwenden wenn eine vollstГӨndige Liste benГ¶tigt wird.

## Frontend-Architektur

SPA aus `index.html`, `style.css` und ES-Modulen вҖ“ **kein Build-Step**, kein Bundler. Alpine.js und vis-network via CDN.

**Modul-Pattern:** Alle `public/js/*.js`-Module (ausser `app.js`, `utils.js`, `prompts.js`, `two-tier.js`) exportieren ein Methoden-Objekt (z.B. `export const lektoratMethods = { ... }`). Die Methoden verwenden `this`, das auf die Alpine-Komponente zeigt вҖ“ per `...spread` in das Alpine-Data-Objekt eingebettet. Neue FunktionalitГӨt: Modul anlegen вҶ’ exportieren вҶ’ in `app.js` spreaden.

**Style-Regel:** Alle Styles ausschliesslich in `style.css` вҖ“ keine Inline-`style`-Attribute, keine `<style>`-BlГ¶cke im HTML.

**Feature-ExklusivitГӨt (Standardverhalten):** Features Гјberschreiben sich gegenseitig вҖ“ es ist immer nur eine Hauptansicht aktiv. Buchebenen-Features (Figuren, Szenen, Ereignisse, Orte, KontinuitГӨt, Buchbewertung, Buchstatistik, Buch-Chat) und Seitenebenen-Features (Lektorat-Editor) sind gegenseitig exklusiv. Jedes Buchebenen-Toggle ruft `_closeOtherMainCards(keep)` in `app.js` auf вҖ“ das schliesst alle anderen Buchkarten und ruft `resetPage()` auf (schliesst Editor + Seiten-Features). `selectPage()` schliesst seinerseits alle Buchkarten bevor es den Editor Г¶ffnet. **Two-Tier-Analyse-Pattern:** `routes/jobs.js` implementiert fГјr Review, Figurenextraktion und Buch-Chat ein Two-Tier-Muster: `SINGLE_PASS_LIMIT` = 60 000 Zeichen вҖ“ darunter Single-Pass direkt an Claude, darГјber Multi-Pass pro Kapitel + Konsolidierungs-Call.

## Komplettanalyse-Job (`komplett-analyse`)

Der Button **В«Alles aktualisierenВ»** startet den einzigen serverseitigen Job `/jobs/komplett-analyse`.

**Pipeline (intern, ein einziger Job):**

```
Phase 1 вҖ“ Vollextraktion kombiniert (parallel pro Kapitel) вҖ“ KEIN separater P5-Call
          buildExtraktionKomplettChapterPrompt()
          System-Prompt: SYSTEM_KOMPLETT_EXTRAKTION (Schema darin eingebettet вҶ’ gecacht)
          Output: figuren + orte + fakten + szenen(Namen) + assignments(Namen)
          вҶ’ Checkpoint 'p1_full_done' (alle 5 Arrays)

Phase 2 вҖ“ Figuren konsolidieren
          buildFiguresBasisConsolidationPrompt()
          вҶ’ saveFigurenToDb() вҶ’ figNameToId aufbauen

Block 1 [parallel] вҖ“
  Phase 3 вҖ“ SchauplГӨtze konsolidieren
            buildLocationsConsolidationPrompt() вҶ’ saveOrteToDb() вҶ’ ortNameToId aufbauen
  Phase 4 вҖ“ Soziogramm (nur wenn вүҘ 4 Figuren)
            buildFigurSoziogrammEnrichmentPrompt()

Block 2 [parallel] вҖ“
  Phase 5 вҖ“ Szenen remappen (kein API-Call)
            chapterSzenen/chapterAssignments aus Phase 1
            Namen вҶ’ IDs via figNameToId / ortNameToId
            вҶ’ figure_scenes + updateFigurenEvents()
  Phase 6 вҖ“ Zeitstrahl konsolidieren
            buildZeitstrahlConsolidationPrompt()
  Phase 8 вҖ“ KontinuitГӨtscheck
            Multi-Pass: buildKontinuitaetCheckPrompt() (Fakten aus Phase 1)
            Single-Pass: buildKontinuitaetSinglePassPrompt()
```

**Token-Optimierungen:** Phase 1 und 5 sind zu einem einzigen Kapitel-Call zusammengefГјhrt. Schema und Regeln leben in `SYSTEM_KOMPLETT_EXTRAKTION` (System-Prompt) und werden gecacht вҖ“ spart ~19/20 Schema-Tokens pro Kapitel-Call. Szenen/Events nutzen Klarnamen; das ID-Remapping erfolgt in `processSzenenEreignisseFromMerged` nach der Konsolidierung.

**Prompt Caching:** `lib/ai.js` setzt `cache_control: { type: 'ephemeral' }` auf allen Systemprompts und sendet den Header `anthropic-beta: prompt-caching-2024-07-31`. Innerhalb eines parallelen Kapitel-Batches (Phase 1) trifft jeder Folge-Call den Cache des Systemprompts inkl. Schema (~10 % des normalen Input-Preises).

**Checkpoint:** `phase: 'p1_full_done'` speichert alle 5 Arrays (`chapterFiguren`, `chapterOrte`, `chapterFakten`, `chapterSzenen`, `chapterAssignments`). Alte Checkpoints mit Phase `p1_done` werden verworfen und der Job neu gestartet.

**UI:** `alleAktualisieren()` in `app.js` startet nur `/jobs/komplett-analyse` und pollt den Fortschritt. Nach Abschluss werden `loadFiguren`, `loadOrte`, `loadSzenen` und `_loadKontinuitaetHistory` parallel neu geladen.
