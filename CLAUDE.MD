## Harte Regeln

- **Prompts nur in `public/js/prompts.js`** вҖ” einzige Quelle fГјr alle Prompt-Schemas und Build-Logik. Server importiert via dynamic `import()`. NIEMALS Prompts in Route-Handlern, Config-Dateien oder anderswo duplizieren.
- **KI-Calls nur via Job-Queue** вҖ” neue Features implementieren einen Job-Typ in `routes/jobs/` (Funktion `runXxxJob` + `router.post`). Direkte synchrone KI-Calls aus Route-Handlern sind verboten. Einzige Ausnahme: Seiten-Chat (`/chat/send`) nutzt bewusst SSE-Streaming.
- **`callAI` gibt nur JSON zurГјck** вҖ” jeder Systemprompt muss JSON-Only erzwingen (`JSON_ONLY`-Konstante in `prompts.js`). Nach jedem `callAI`-Aufruf Pflichtfeld prГјfen (z.B. `fehler`, `gesamtnote`, `figuren`). Fehler werfen statt falsche Daten rendern.
- **Styles nur in `style.css`** вҖ” keine Inline-`style`-Attribute, keine `<style>`-BlГ¶cke im HTML.
- **`bsGetAll` statt `bsGet` fГјr Listen** вҖ” BookStack paginiert (Standard 20 EintrГӨge). `bsGetAll` iteriert alle Seiten automatisch.
- **401-Handling zentral** вҖ” ein globaler `window.fetch`-Wrapper in `public/js/app.js` fГӨngt alle 401-Antworten ab und dispatcht `session-expired`; Alpine zeigt daraufhin den Session-Banner. Feature-Module prГјfen 401 nicht selbst und dГјrfen das Event nicht unterdrГјcken. Kein Auto-Redirect вҖ“ User soll ungespeicherte Inhalte retten kГ¶nnen.

## Neues Feature hinzufГјgen

1. Job-Datei in `routes/jobs/` anlegen (Pattern: siehe `routes/jobs/review.js`)

4. Prompt-Builder in `public/js/prompts.js` ergГӨnzen

### Frontend

1. Neues Modul in `public/js/` вҶ’ Methoden-Objekt exportieren (z.B. `export const xxxMethods = { ... }`)
2. Methoden verwenden `this` (zeigt auf Alpine-Komponente)

### Feature-Toggle (ExklusivitГӨt)

- Jedes Buchebenen-Toggle ruft `_closeOtherMainCards(keep)` auf (schliesst alle anderen Karten + Editor)
- `selectPage()` schliesst alle Buchkarten bevor der Editor Г¶ffnet
- Neues Feature-Toggle immer nach diesem Muster implementieren

## Prompt-System

**Zwei Dateien, klare Trennung:**
- `prompt-config.json` (Projektroot, Pflichtdatei) вҖ” Rollenformulierungen, Basisregeln, Buchtypen pro Sprache. Fehlt sie вҶ’ Server-Crash beim Start.
- `public/js/prompts.js` вҖ” JSON-Schemas, Build-Logik, `configurePrompts()`. Wird sowohl vom Server (dynamic `import()`) als auch vom Frontend (ESM) geladen.

**Ladereihenfolge:**
- Server: `routes/jobs.js` und `routes/chat.js` lesen `prompt-config.json` synchron beim Modulstart вҶ’ `configurePrompts()` einmalig. `routes/proxies.js` liefert die Config lazy beim ersten `/config`-Call ans Frontend.
- Frontend: `app.js` вҶ’ `init()` вҶ’ `configurePrompts(cfg.promptConfig)` вҶ’ setzt `SYSTEM_*`-Variablen via ESM-Live-Binding.

**Buchtypen:** In `prompt-config.json` unter `buchtypen`, aufgeteilt nach Sprachcode (`de`, `en`). Jeder Key hat `label` + `zusatz`. Neuer Typ: in beiden Sprachen ergГӨnzen.

**Per-Buch-Kontext:** `getBookPrompts(bookId)` вҶ’ `getLocalePromptsForBook()` augmentiert `baseRules` dynamisch mit Buchtyp-Zusatztext (`BUCHTYP-KONTEXT:`) und Freitext des Users (`VORRANGIGE ANGABEN DES AUTORS:` вҖ“ Гјbersteuert bei Konflikt die Basisregeln, insbesondere Stil/Ton/Format).

## Datenbank

Schema, Tabellen und Migrationslogik: siehe [db/schema.js](db/schema.js).

**Migration hinzufГјgen:** Neuen `if (version < N)`-Block in `runMigrations()` ergГӨnzen (N = nГӨchste fortlaufende Nummer, aktuell bei 40) + `UPDATE schema_version SET version = N`. Neue Tabellen als `CREATE TABLE IF NOT EXISTS` вҖ” keine Versionierung nГ¶tig.

**Neuer Beziehungstyp:** Keine SchemaГӨnderung. `figure_relations.typ` ist Freitext. Neuen Typ in der `BZ`-Konstante (Frontend-Rendering) und im Claude-Prompt (`FINAL_SCHEMA` in `prompts.js`) ergГӨnzen.

## KI-Provider

Drei Provider, konfiguriert via `API_PROVIDER` in `.env`:

| Provider | Env-Vars | Besonderheit |
|----------|----------|--------------|
| `claude` | `ANTHROPIC_API_KEY`, `MODEL_NAME` | Prompt-Caching (`cache_control: ephemeral`), grosses Kontextfenster |
| `ollama` | `OLLAMA_HOST`, `OLLAMA_MODEL`, `OLLAMA_TEMPERATURE` | Mutex-Serialisierung (VRAM-Schutz), dynamische `num_ctx`-Berechnung |
| `llama` | `LLAMA_HOST`, `LLAMA_MODEL`, `LLAMA_TEMPERATURE` | llama.cpp, ebenfalls Mutex-serialisiert |

**`MODEL_TOKEN`** setzt den globalen Output-Token-Cap (`MAX_TOKENS_OUT` in `lib/ai.js`, Default 64 000). Job-spezifische Overrides werden per `Math.min` gedeckelt.

**JSON-Parsing:** `lib/ai.js` hat mehrstufigen Fallback: `JSON.parse()` вҶ’ `extractBalancedJson()` вҶ’ `jsonrepair()`.

## Two-Tier-Analyse

Jobs in `routes/jobs/` verwenden ein Single-Pass/Multi-Pass-Muster. Limits und Batch-GrГ¶ssen sind als Konstanten in `routes/jobs/shared.js` definiert вҖ” dort nachschlagen statt hier. FГјr `komplett-analyse` gilt ein hГ¶heres Limit bei Claude (passt ins 200K-Token-Kontextfenster).

## Komplettanalyse-Job

**Pipeline-Phasen und AbhГӨngigkeiten:**

```
Phase 1 вҖ“ Vollextraktion (parallel pro Kapitel oder Single-Pass)
          вҶ’ figuren, orte, fakten, szenen(Namen), assignments(Namen)
          вҶ’ Checkpoint 'p1_full_done'
                    вҶ“
Phase 2 вҖ“ Figuren konsolidieren + Soziogramm (aus P2-Output, kein Extra-Call)
Phase 3 вҖ“ SchauplГӨtze konsolidieren
Phase 3b вҖ“ KapitelГјbergreifende Beziehungen (nur Multi-Pass, non-critical)
                    вҶ“
Block 2 [parallel]:
  Phase 5 вҖ“ Szenen remappen (kein API-Call, Namen вҶ’ IDs)
  Phase 6 вҖ“ Zeitstrahl konsolidieren
  Phase 8 вҖ“ KontinuitГӨtscheck (Single-Pass: voller Text, Multi-Pass: Fakten)
```

**Standalone-KontinuitГӨtscheck:** `POST /jobs/kontinuitaet` вҖ” lГӨuft Phase 8 einzeln, ohne die volle Pipeline. Exportiert `runKontinuitaetJob` aus `routes/jobs/komplett.js`.

**Wichtige Mechanismen:**
- **Delta-Cache:** Phase 1 (Multi-Pass) prГјft `chapter_extract_cache` in der DB. Cache-Key enthГӨlt `pages_sig` (sortierte `page_id:updated_at`-Paare). Г„ndert sich eine Seite вҶ’ Cache-Miss вҶ’ Neu-Extraktion. Single-Pass wird nicht gecacht.
- **Prompt-Caching:** System-Prompt mit eingebettetem Schema wird bei parallelen Kapitel-Calls gecacht (~10% des Input-Preises fГјr Folge-Calls).
- **Checkpoint-Wiederaufnahme:** `p1_full_done` speichert alle 5 Arrays. Alte `p1_done`-Checkpoints werden verworfen вҶ’ Job-Neustart.

## Chat

- **Seiten-Chat** (`/chat/send`): SSE-Streaming, kein Job-Queue. Antwortformat enthГӨlt `vorschlaege` mit zeichengenauem `original` fГјr Textersetzung.
- **Buch-Chat** (`/jobs/book-chat`): Job-Queue, kein VorschlГӨge-System. Sessions nutzen `page_id=0`, `page_name='__book__'`.
- **SSE-Fehler:** `sseStarted`-Flag trennt Pre-Stream-Fehler (вҶ’ JSON 502) von Mid-Stream-Fehler (вҶ’ SSE `{ type: 'error' }` + `[DONE]`).

## Fehlerbehandlung

- **Jobs:** `try/catch` вҶ’ `failJob(id, err)` setzt Status auf `'error'` oder `'cancelled'` (bei `AbortError`). Fehler werden in `job.error` gespeichert und geloggt.
- **API-Routen:** Fehlende Parameter вҶ’ `400 JSON`, unauthentifiziert вҶ’ `401 JSON`.
- **JSON-Parsing:** Mehrstufiger Fallback in `lib/ai.js` (siehe KI-Provider).
- **DB-Fehler:** Geloggt, blockieren nicht den Request.

## Logging

Winston (`logger.js`): Level `info`, Ausgabe in `lektorat.log` (5 MB, 3 Dateien rotiert) + Console. Jobs nutzen Child-Logger mit Kontext: `logger.child({ job, user, book })` вҶ’ Format: `[INFO][lektorat|user@mail.com|42] Nachricht`.
